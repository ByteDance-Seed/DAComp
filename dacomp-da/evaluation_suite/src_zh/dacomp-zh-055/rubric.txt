## 需求1：构建可控分析基表并识别错配客户（本需求最高可得 6分）
### 标准1.1 [最高6分]：错配识别数据底座搭建
#### 路径1.1.A [6分｜邮箱归并 + 标准化 + 分位筛选]
- 小标准1.1.A.1 [1分｜完备性]：说明以 `customer360__customer_value_analysis` 为主表，按 `LOWER(primary_email)=LOWER(email)` 归并 `total_sales_amount`（覆盖 5,001/5,001 行），并筛得三项关键指标齐全的 3,361 条记录。
- 小标准1.1.A.2 [4分｜精确性]：给出可复现伪代码并校验锚点：
  ```sql
  total_sales_clamped = clamp(total_sales_amount, 151.62, 3,564,026.72)
  sales_norm   = (total_sales_clamped - 151.62) / (3,564,026.72 - 151.62)
  adopt_norm   = (product_adoption_rate - 0.302) / (1.0 - 0.302)
  support_norm = (support_resolution_efficiency - 0.2978) / (0.98 - 0.2978)
  composite    = 0.40*sales_norm + 0.35*adopt_norm + 0.25*support_norm
  ```
  并基于 `investment_priority_score` 的 CUME_DIST≥0.7、`composite` 的 CUME_DIST≥0.5 取阈值；需验证：`ip_cutoff = 1.63 ±0.01`、`perf_cutoff = 0.40408 ±0.0005`、`top30_count = 1,013 ±3`、`flagged_count = 494 ±2`、错配率 `≈0.4877`。
- 小标准1.1.A.3 [1分｜结论性]：总结邮箱路径实现 100% 覆盖、稳定产出 3,361 条可分析样本，并指出 Top30% 投资客户中约 48.8% 被判定为错配，为后续分析奠定一致口径基表。

#### 路径1.1.B [6分｜多 ID 优先映射 + 组织聚合]
- 小标准1.1.B.1 [1分｜完备性]：声明引入 `customer360__mapping`，按优先级 `zendesk_user_id → marketo_lead_id → stripe_customer_id` 选取匹配；若同一条目存在多行，则先取 `is_organization_header=1`，否则取 `customer360_id` 字典序最小值，并在对接 `customer360__customer` 前完成 `customer360_id` 级聚合。
- 小标准1.1.B.2 [4分｜精确性]：复核关键指标：`resolved_rows = 5,001`、`distinct customer360_id = 4,787 ±5`、`valid_rows = 3,361`，同一归一化流程下得到 `ip_cutoff = 1.63 ±0.01`、`perf_cutoff = 0.40258 ±0.0005`；`top30_count = 1,013 ±3`、`flagged_count = 496 ±2`，阶段错配率 `≈0.4896`。
- 小标准1.1.B.3 [1分｜结论性]：指出多 ID 映射可跨系统绑定 4,787 个唯一客户身份，虽 Top30 规模未变，但错配数增至 496（约 48.9%），揭示组织级重复带来的轻微偏差。

#### 路径1.1.C [6分｜域名回退 + 组织头补齐]
- 小标准1.1.C.1 [1分｜完备性]：说明提取 10 个有效邮箱域名作为回退键，将 `primary_organization` 与 `customer360__mapping.customer360_organization_id` 关联，缺失时使用同域聚合的 `total_sales_amount`（共覆盖 5,001/5,001 行）。
- 小标准1.1.C.2 [4分｜精确性]：在域名层聚合后复用路径 A 标准化流程，验证：全部可回填样本仍得 `valid_rows = 3,361`、`flagged_count = 494 ±2`，阈值保持 `ip_cutoff ≈1.63`、`perf_cutoff ≈0.4041`，说明软约束不破坏筛选结果。
- 小标准1.1.C.3 [1分｜结论性]：强调域名/组织头回退保证跨邮箱别名场景不丢失客户，错配规模与路径 A/B 等价，可作为邮箱路径的兜底策略。

## 需求2：ROI 量化与生命周期分布解析（本需求最高可得 6分）
### 标准2.1 [最高6分]：错配客户 ROI 画像
#### 路径2.1.A [6分｜Winsorize + 生命周期层析]
- 小标准2.1.A.1 [1分｜完备性]：说明在路径1.1.A的 494 个错配客户内计算 `ROI = CLV / acquisition_cost`，对 `acquisition_cost ≤0` 置为 NULL（本样本无负值），并对 ROI 做 P1/P99 Winsorize 处理。
- 小标准2.1.A.2 [4分｜精确性]：输出关键统计：`p1 = 1.2297 ±0.02`、`median = 6.768 ±0.05`、`p99 = 39.724 ±0.5`、`mean = 8.738 ±0.05`、`ROI>1 占比 = 99.60% ±0.1pp`；生命周期分布 `{Maturity:208, Retention:132, Growth:131, Activation:17, Dormant:6}`，对应均值 `{9.145, 8.161, 8.428, 9.089, 13.124}`。
- 小标准2.1.A.3 [1分｜结论性]：指出错配客户 92% 聚焦在 Maturity/Retention/Growth，整体 ROI 虽高于 1，但相较匹配组均值低 1.93pt，反映投资优先级与产出脱节。

#### 路径2.1.B [6分｜对数 ROI + 方差检验]
- 小标准2.1.B.1 [1分｜完备性]：说明使用 Winsor 化 ROI 取 `log_roi = ln(ROI)`，并限定样本为路径1.1.A错配集的 `{Growth, Maturity, Retention}` 三阶段。
- 小标准2.1.B.2 [4分｜精确性]：报告 `mean(log_roi) = 1.913 ±0.01`、`std = 0.715 ±0.01`；执行单因素 ANOVA 得 `F = 1.98`、`p = 0.139`（>0.05，不显著），并列出阶段均值差异：`Growth` 比 `Maturity` 低 `0.106`，高于 `Retention` `0.042`。
- 小标准2.1.B.3 [1分｜结论性]：说明目前生命周期间 log ROI 差异未达显著水平，提示需结合其他画像特征才能解释 ROI 波动。

#### 路径2.1.C [6分｜ROI 分位段 × 生命周期矩阵]
- 小标准2.1.C.1 [1分｜完备性]：基于错配集按 ROI 构建 `Q1/Q2/Q3`（阈值 `4.84`、`9.22`），与生命周期组合成至少 3×3 矩阵。
- 小标准2.1.C.2 [4分｜精确性]：矩阵需列示每格样本量与中位数，例如 `Growth` 中 `{Q1:53, median 3.90}`, `{Q2:35, median 6.58}`, `{Q3:43, median 12.82}`；`Maturity`、`Retention`、`Activation`、`Dormant` 同步给出。并标注各阶段错配率与路径1.1结果一致（如 `Growth ≈0.5325`、`Maturity ≈0.4674` 等）。
- 小标准2.1.C.3 [1分｜结论性]：强调即使进入高 ROI 分位（Q3）仍普遍存在错配（如 Maturity Q3 77/208），证明现行优先级模型未能过滤生命周期风险。

## 需求4：行为机制验证（本需求最高可得 6分）
### 标准4.1 [最高6分]：入职质量、团队结构与效果联动
#### 路径4.1.A [6分｜分组相关 + 显著性]
- 小标准4.1.A.1 [1分｜完备性]：分别在错配组(494)与非错配组(519)计算 `corr(onboarding_score, product_adoption_rate)`。
- 小标准4.1.A.2 [4分｜精确性]：报告 `r_flagged = 0.293`、`t = 6.79`、`p = 3.31e-11`；`r_nonflag = 0.287`、`t = 6.81`、`p = 2.79e-11`。
- 小标准4.1.A.3 [1分｜结论性]：指出两组均呈显著正相关，错配客户对入职质量更敏感，应作为修复错配的首要杠杆。

#### 路径4.1.B [6分｜分桶对比]
- 小标准4.1.B.1 [1分｜完备性]：按 `customer_onboarding_score` 划分 `<2.5 / 2.5-3.5 / ≥3.5`，分别统计错配与非错配的采用率与 ROI。
- 小标准4.1.B.2 [4分｜精确性]：输出均值差异：错配组平均入职 `2.806`、采用率 `0.666`、ROI `8.80`；非错配组 `3.187`、`0.852`、`10.73`。各分段示例：
  - 错配 `<2.5`：`n=157`，采用率 `0.628`，ROI `7.99`；
  - 错配 `≥3.5`：`n=64`，采用率 `0.696`，ROI `9.30`；
  - 非错配 `2.5-3.5`：`n=296`，采用率 `0.845`，ROI `11.21` 等。
- 小标准4.1.B.3 [1分｜结论性]：说明入职评分越高，错配 vs 非错配的落差越小，建议把入职质量作为进入高投资池的最低门槛。

#### 路径4.1.C [6分｜团队规模 × 决策层级]
- 小标准4.1.C.1 [1分｜完备性]：构建团队规模桶 `Small ≤10 / Medium 11-50` 与 `Exec/NonExec` 组合（样本≥15）。

  - `Small+Exec`：错配 `7.66` vs 非错配 `8.66`；
  - `Small+NonExec`：`8.55` vs `9.67`；
  - `Medium+Exec`：`14.63` vs `14.96`；
  - `Medium+NonExec`：`14.11` vs `13.65`；同时报告相关系数 `corr(team_size, ROI)`：错配 `0.317`、非错配 `0.276`。
- 小标准4.1.C.3 [1分｜结论性]：指出“小团队+高层参与”虽有提升但不足以消除错配，中型团队表现最优，建议在优先级模型中引入团队规模与决策层级的交互项。

## 需求5：诊断投资决策偏差并提出优化方案（本需求最高可得 6分）
### 标准5.1 [最高6分]：偏差识别与改进路径
#### 路径5.1.A [6分｜偏差诊断 + 证据链]
- 小标准5.1.A.1 [1分｜完备性]：覆盖三项诊断：`corr(investment_priority, composite) = 0.0319`、`corr(investment_priority, ROI) = 0.0090`、关键画像错配率（如 `Growth 0.533`、`Dormant 0.750`、`Education+Small 0.826`、`Technology+Startup 0.800`）。
- 小标准5.1.A.2 [4分｜精确性]：给出投资优先级三分位均值 `{T1:0.4047, T2:0.4006, T3:0.4135}`（呈非单调），并引用高错配组合的 RR 或阶段错配率，量化模型误差及 ROI 均值差 `ΔROI = 1.93`（匹配 10.73 vs 错配 8.80）。
- 小标准5.1.A.3 [1分｜结论性]：总结当前模型过度依赖静态画像，忽视生命周期、入职质量与团队结构，必须进行系统性重构。

#### 路径5.1.B [6分｜特征对齐度与惩罚机制]
- 小标准5.1.B.1 [1分｜完备性]：定义 `alignment = corr(feature, investment_priority) × corr(feature, composite)` 并筛选对齐度为负的特征。
- 小标准5.1.B.2 [4分｜精确性]：给出示例：
  - `customer_onboarding_score`：`corr_ip = -0.017`、`corr_perf = 0.314`、alignment `≈-0.0053`；
  - `team_size`：`corr_ip = 0.018`、`corr_perf = 0.316`、alignment `≈0.0057`；
  - `decision_maker_level (Exec=1)`：`corr_ip = 0.0265`、`corr_perf = -0.138`、alignment `≈-0.0037`。并提出对应加权/惩罚公式（如对负对齐特征增加惩罚因子、对 team_size 给予正向加权）。
- 小标准5.1.B.3 [1分｜结论性]：指出负对齐特征（低入职、高层信号）导致系统性错配，需引入惩罚项或联动阈值（如“Exec 信号需伴随采用率≥0.80”）。

#### 路径5.1.C [6分｜运营 + 模型双轨建议 ≥6条]
- 小标准5.1.C.1 [1分｜完备性]：提出至少 6 条涵盖模型校准、阶段阈值、行业差异化、入职/支持流程与监控的建议。
- 小标准5.1.C.2 [4分｜精确性]：每条建议绑定量化目标，例如：
  1. “针对 Education/Tech Startup 客户提升入职评分至 ≥3.0，预计错配率从 0.8 降至 ≤0.55”；
  2. “Growth 阶段纳入 `adoption≥0.75` 的优先级闸门，预计错配率下降 ≥8pp”；
  3. “引入 ROI 下限：Top30 客户 ROI 中位数≥9，否则自动降级”；
  4. “对 team_size≤10 且 Exec=1 的客户附加 -0.1 权重，平衡过度配置”；
  5. “行业小组设支持效率 KPI ≥0.82，优先治理 Retail/Healthcare 小型客户”；
  6. “上线季度错配监控仪表，若阶段错配率>0.5 自动触发入职/支持专项优化”。
- 小标准5.1.C.3 [1分｜结论性]：形成路线图：先模型校准→差异化运营→持续监控，并设定闭环 KPI（如错配率降低 ≥30%、错配客户 ROI 提升 ≥15%、高风险组合 RR 收敛至 ≤1.2）。
