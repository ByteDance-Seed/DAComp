## Requirement 1: Consistent data definitions, traceable metrics, and risk quantification layering (S→PD→LGD→Regulatory Parameters) (Maximum 22 points)
### Standard 1.1: End-to-end definitions and re-computability (Maximum 4 points)
#### Path 1.1.A (Field mapping + aggregation definitions + templates + audit trail)
- Sub-criterion 1.1.A.1 [1 point | Completeness] Field mapping and transformation notes  
  Requirements: Provide source table/source field → standard field mappings, noting units, types, and cleansing/derivation rules (e.g., missing handling, outlier treatment, currency/tax rate conversions).  
  Acceptable evidence: Field dictionary tables, annotated SQL/code blocks, data flow diagrams.  
  Scoring conditions: Mappings cover core input domains (revenue, stability, dependency, existing Credit Rating/alternatives); transformation logic is clear and traceable back to source fields.
- Sub-criterion 1.1.A.2 [1 point | Accuracy] Core aggregation definitions and sample re-computation  
  Requirements: Clarify the calculation definitions for “enterprise-level” aggregation (e.g., time window, de-duplication rules, inclusive/exclusive of tax definitions), and demonstrate at least one “sample chain” from raw rows → statistics recalculation.  
  Acceptable evidence: Calculation formulas + example SQL/reconciliations from detail → aggregation → result tables/screenshots.  
  Scoring conditions: Formulas align with implementation; the sample chain is closed-loop (inputs, calculations, outputs consistent); key definitions (time, whether to exclude red-letter reversals/voided or canceled records, etc.) are explained.
- Sub-criterion 1.1.A.3 [1 point | Conclusiveness] Recomputable templates and intermediate quantities  
  Requirements: Provide repeatable templates (SQL/scripts/Excel workflows) whose outputs include key intermediate quantities (e.g., monthly flows, CV, Top1/HHI).  
  Scoring conditions: Running the same inputs multiple times yields consistent results; templates run independently (dependencies and parameters are well-documented); key intermediate quantities are visible in outputs.
- Sub-criterion 1.1.A.4 [1 point | Completeness] Versioning and audit trail  
  Requirements: Definition statements, version numbers (or Git tags/script versions), change logs, and run logs (time window/input file names/record counts).  
  Scoring conditions: Version and definition changes are traceable; logs can pinpoint a specific run (including inputs, time, output row counts/summaries).

### Standard 1.2: Composite risk score S (Maximum 5 points)
#### Path 1.2.A (Weighted aggregation and validation)
- Sub-criterion 1.2.A.1 [1 point | Completeness] Dimensions, weights, and directionality  
  Requirements: List the S indicators, weights (sum = 100%), and directionality (risk/protective factors; protective factors use reverse scoring or deductions).  
  Scoring conditions: Each indicator’s contribution direction aligns with business meaning; total weights are correct; treatment of protective factors is clear.
- Sub-criterion 1.2.A.2 [1 point | Accuracy] Monotonicity validation  
  Requirements: Using boundary examples/unit tests, show that when a risk factor worsens (or a protective factor declines), S rises rather than falls; and vice versa.  
  Scoring conditions: Provide at least one example each for two factor types (risk/protective) proving S’s response direction is correct; when conflicts arise, provide fixes (e.g., truncation/Winsorization/log transformations).
- Sub-criterion 1.2.A.3 [1 point | Conclusiveness] S intervals and human consistency  
  Requirements: Define good/medium/weak S intervals or quantile thresholds; sample and compare consistency against human judgment or existing Credit Ratings.  
  Scoring conditions: Interval segmentation has justification (distributions, business thresholds); provide sampling methods and conclusions (e.g., consistency ratio or explanations of typical mismatches).
- Sub-criterion 1.2.A.4 [1 point | Accuracy] Stability/drift tests  
  Requirements: Compare S distributions over rolling periods (or KS/PSI/equivalent statistics); when anomalies appear, provide causes and corrections (e.g., re-calibration, definition harmonization).  
  Scoring conditions: Provide testing process and results; give reasonable explanations and treatment strategies for threshold exceedances.
- Sub-criterion 1.2.A.5 [1 point | Completeness] Reproducibility and versioning  
  Requirements: S computation can be re-run; provide version numbers, dependencies, parameters, etc.; if models are involved, set random seeds.  
  Scoring conditions: Repeated runs are consistent; versions, dependencies, and parameters are clear for audit.

## Requirement 2: Single-customer pricing and limit calculation → Portfolio allocation and churn-constrained optimization (Maximum 22 points)

  Requirements: r = CoF + OpEx + PD×LGD + K×ROE_min + Margin (or equivalent mapping), with clear parameter definitions and sources.
